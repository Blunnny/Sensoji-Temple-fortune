# 赛博占卜网站路线图

本路线图围绕「浅草寺抽签 × 摄像头 × 手势操作」这一核心概念，从零开始规划整个项目的实施步骤。后续我们可以按阶段逐步实现，并在需要时迭代调整。

---

## 0. 项目概述与目标

- **核心体验**
  - 用户打开网页 → 允许摄像头 → 通过手势进行「抽签 / 切换 / 确认」等操作。
  - 抽到的签以和风纸张样式展示，并附有签文原文、翻译与解读。
- **设计原则**
  - 优先保证「体验顺滑」和「仪式感」，再逐步增强功能。
  - 手势交互是增强，而不是唯一入口，保留备用按钮操作以供用户选择。
- **阶段性目标**
  1. 能正常「抽到一支签」并展示内容。
  2. 能通过摄像头识别基本手势，完成抽签操作。
  3. 手势稳定、UI 有和风质感、内容体系丰富。

---

## 1. 技术选型与项目骨架

- **前端框架**
  - React + Vite + TypeScript。
- **手势识别**
  - 优先选择浏览器端已有方案：
    - MediaPipe Hands（通过 WASM / JS）或
    - TensorFlow.js 手势识别示例模型。
  - 要求：能提供手部关键点或手势类别，支持本地推理、无需上传视频。
- **项目初始化**
  - 创建前端项目骨架（含基础路由 / 状态管理方案）。
  - 约定代码结构：组件目录、hooks、手势识别相关封装、数据文件等。

> 本阶段产出：一个空壳前端项目，可以跑起来看到简单页面，但暂时没有抽签和手势功能。

---

## 2. 签文内容结构与数据设计

- **数据结构设计**
  - 为每一支签定义统一结构（示例字段）：
    - `id`: 签编号（如 `1`、`第1签` 等）
    - `title`: 签题（如「七宝浮図頂」）
    - `grade`: 运势级别（大吉 / 吉 / 凶 等）
    - `original`: 原文（日文/汉文）
    - `reading`: 读音或假名（可选）
    - `translation`: 现代语/中文翻译
    - `fortunes`: 各方面解读（恋爱 / 工作 / 健康 / 旅行 等，可用对象或数组）
    - `notes`: 其他补充说明（可选）
- **数据存储方式**
  - MVP 阶段使用前端本地 JSON 文件，后续若需要再迁移到后端数据库。
  - 支持按 `id` 访问、随机抽取、根据运势筛选等基础操作。
- **收集与录入内容**
  - 先录入少量代表性签文用于开发调试（例如 10~20 支）。
  - 后续再逐步扩充到完整签文集合。

> 本阶段产出：`omikuji.json` 等数据文件，以及读取与抽签的基础工具函数。

---

## 3. 基础抽签流程（无摄像头与手势）

- **目标**
  - 在不考虑摄像头和手势的前提下，先实现「普通网页抽签」的完整流程，确保业务逻辑清晰可靠。
- **功能点**
  - 抽签按钮：点击后随机抽取一支签。
  - 签面展示：
    - 显示签的编号、题目、运势、原文、翻译与解读。
    - 简单的「和纸 / 竹筒」风格 UI。
  - 历史记录（可选）：
    - 使用本地存储（localStorage）记录用户最近抽到的几支签。
- **结构分层**
  - 抽签逻辑封装为独立模块（如 `useOmikuji` hook 或 service），与 UI 解耦。
  - 方便后续手势操作直接复用同一套抽签逻辑。

> 本阶段产出：一个无需摄像头、仅用鼠标/触摸即可玩的占卜网页。

---

## 4. 摄像头接入与基础调试

- **摄像头权限与预览**
  - 使用 `getUserMedia` 请求视频流权限，并在页面上展示缩略预览窗口。
  - 加入权限被拒绝 / 无摄像头设备时的提示与降级路径（继续用按钮操作）。
- **性能与资源控制**
  - 限制视频分辨率与帧率（例如 640×480，15–30fps）。
  - 在适当时机停止/重启视频流，避免占用过多资源。
- **用户引导**
  - 在界面上显示「请把手伸到画面中间」「保持光线充足」等文字/图示。

> 本阶段产出：页面中可以看到实时摄像头预览，并能稳定获取视频流帧。

---

## 5. 手势识别 MVP：握拳抽签

- **目标**
  - 实现一个最小可行的手势交互：用户握拳 → 触发一次抽签。
- **步骤**
  - 集成 MediaPipe Hands 或其他手势识别库：
    - 从视频流中获取手部关键点数据。
    - 在每帧计算当前手势状态（如「张开」「握拳」）。
  - 实现「握拳」判定规则：
    - 根据指尖相对于关节的位置判断各手指是否弯曲。
    - 当所有手指都弯曲到一定阈值以内，判定为握拳。
  - 加入状态稳定逻辑：
    - 避免抖动导致连续触发多次抽签，可加「冷却时间」或「需保持握拳若干帧」。
- **反馈与可视化**
  - 在 UI 上增加状态提示（如「检测到握拳，即将抽签」）。
  - 可选：在视频预览上绘制手部骨架，方便调试。

> 本阶段产出：用户通过握拳即可完成抽签，体验到「赛博摇签」的第一步。

---

## 6. 扩展手势：左右切换与指向确认

- **左右移动切换签**
  - 通过手部中心点在 X 轴方向的位移、速度判断「左挥」/「右挥」。
  - 配合冷却时间与阈值，减少误触发。
  - 行为：左挥 = 上一签，右挥 = 下一签，或用于切换不同类别（恋爱 / 工作等）。
- **食指指向/点击选择**
  - 判定仅有食指伸出，其余手指弯曲。
  - 将食指指尖映射到画面坐标，判断是否在某个按钮区域内。
  - 可通过「停留一定时间」或「伸出→收回」来触发“点击”。
- **多模态容错**
  - 保留屏幕按钮，用户可以任意选择手势或传统点击。
  - 对识别失败或不稳定时给出文字提示与退路。

> 本阶段产出：手势交互覆盖「抽签」「切换」「确认」等主要操作，形成完整的体感操作体验。

---

## 7. 视觉风格与动效打磨

- **和风视觉设计**
  - 选择合适的和风配色、纸张纹理背景、毛笔或仿宋类字体。
  - 签面排版参考真实浅草寺签文的布局。
- **动效与过场**
  - 抽签过程可加入「摇签筒 → 掉出一支签」的动画。
  - 抽到签时加入轻微缩放、淡入淡出、光效等，增强仪式感。
- **响应式布局**
  - 保证在桌面端和手机竖屏上都能良好显示。
  - 适配不同比例摄像头画面与操作区域。

> 本阶段产出：一个在视觉和交互上都「有质感」的在线占卜体验。

---

## 8. 部署、分享与后续扩展

- **部署与访问**
  - 使用静态托管服务（如 Vercel、Netlify、GitHub Pages 等）发布网站。
  - 配置 HTTPS 以确保摄像头功能可用。
- **分享与增长**
  - 生成社交分享图/文案，让用户方便分享「今日抽到的是第 X 签」。
  - 可记录匿名统计数据（抽到某签的频率等）用于后续调优。
- **后续扩展方向**
  - 多语言支持（中文 / 日文 / 英文）。
  - 用户登录与个人占卜记录。
  - 更多占卜形式（如大运势、合盘、每日一签推送等）。

---

## 9. 里程碑划分建议

1. **MVP 1：纯网页抽签版**
   - 完成签文数据结构与基础抽签 UI。
   - 不含摄像头与手势。
2. **MVP 2：摄像头 + 握拳抽签**
   - 集成手势识别库，实现握拳抽签。
   - 提供基础视觉体验。
3. **MVP 3：完整手势交互版**
   - 加入左右切换、指向操作。
   - 优化误判、增加指示与反馈。
4. **MVP 4：视觉与内容增强版**
   - 大幅完善签文内容与翻译。
   - 打磨和风 UI、动效、分享体验。

> 在实际开发中，我们可以先完成 MVP 1，随后按阶段增加功能，每个阶段都可独立上线、收集反馈并迭代。
